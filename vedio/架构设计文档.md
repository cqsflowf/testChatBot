## 一、系统概述

### 1.1 设计目标

设计一个**全双工语音对话系统 v3.5**，具备以下核心能力：

1. **实时语音采集与处理**：从麦克风采集音频流
2. **声学VAD**：使用Silero VAD检测用户语音的起止与打断，阈值500ms
3. **Qwen ASR 17B**：使用Qwen ASR 17B模型将语音转换为文本，帧长20ms
4. **Qwen3.5-Flash语义VAD**：流式判断语义完整性 + 有效人声判断
5. **并行情绪识别**：使用Qwen3 Omni Flash模型识别用户情绪
6. **融合阶段**：文本和情绪同时输入LLM进行任务规划
7. **LLM流式输出**：LLM响应实时流式输出到前端显示
8. **流式TTS播报**：按句子实时转换语音，边生成边播报
9. **工具调用**：支持SKILLS和MCP协议的工具调用
10. **结果总结**：LLM对工具调用结果进行总结
11. **全双工交互**：语义VAD判断有效人声 → 立即停止TTS → 继续接收完整句子
12. **多轮对话**：支持上下文，对话连贯
13. **时延监控**：实时追踪各模块处理时延，便于性能分析
14. **并发处理** (v3.5新增)：音频接收与处理解耦，非阻塞架构

### 1.2 声学VAD与语义VAD分工

| 类型 | 职责 | 模型 |
|------|------|------|
| **声学VAD (aVAD)** | 听**声音有没有人在说话**：人声起点检测、静音检测、噪声过滤、触发打断确认 | Silero VAD |
| **语义VAD (sVAD)** | 听**话有没有说完**：语义完整性判断、有效人声判断、语气助词过滤 | Qwen3.5-Flash |

**全双工核心 = aVAD 快 + sVAD 准 + 可打断**

### 1.3 v3.5 核心改进

#### 并发架构改进（v3.5 最重要的改进）

```
┌─────────────────────────────────────────────────────────────────┐
│                    v3.5 并发架构                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  前端音频流 (实时)                                               │
│       │                                                         │
│       ▼                                                         │
│  process_audio() - 非阻塞放入队列                               │
│       │                                                         │
│       ▼                                                         │
│  asyncio.Queue (音频队列)                                       │
│       │                                                         │
│       ▼                                                         │
│  _audio_processing_loop() - 后台循环处理                        │
│       │                                                         │
│       ├──── VAD/ASR/情绪识别 (并行)                             │
│       │                                                         │
│       └──── LLM推理 (后台任务，不阻塞)                           │
│                                                                 │
│  关键优势：                                                      │
│  - 音频接收与处理解耦，前端不再阻塞                              │
│  - LLM推理作为后台任务运行                                       │
│  - 支持在THINKING状态下实时打断                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**关键改进**：
- **非阻塞接收**：音频放入队列立即返回，不阻塞前端
- **后台处理循环**：独立任务持续处理音频队列
- **LLM后台任务**：LLM推理不阻塞音频接收
- **THINKING打断**：允许在LLM推理期间打断

#### 打断逻辑优化（v3.2-v3.5演进）

```
┌─────────────────────────────────────────────────────────────────┐
│                    v3.5 打断流程（语义驱动）                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  TTS正在播报 或 LLM推理中 (SPEAKING/THINKING)                   │
│       │                                                         │
│       ▼                                                         │
│  声学VAD检测到人声                                               │
│       │                                                         │
│       ▼                                                         │
│  启动打断确认模式（不立即停止TTS）                               │
│       │                                                         │
│       ▼                                                         │
│  启动ASR流式识别（打断模式）                                     │
│       │                                                         │
│       ▼                                                         │
│  语义VAD判断人声有效性                                           │
│       │                                                         │
│       ├──── 语气助词(嗯/啊/呃) ──→ 取消打断，继续播报            │
│       │                                                         │
│       ├──── 噪声(咳嗽/环境音) ──→ 取消打断，继续播报             │
│       │                                                         │
│       ├──── 有效人声 ──→ 立即停止TTS/LLM                        │
│       │                   继续接收用户输入                       │
│       │                                                         │
│       └──── 语义完整 ──→ 进入LLM处理                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**关键改进**：
- **支持THINKING状态打断**：v3.5扩展到THINKING状态，允许在LLM推理期间打断
- **语义VAD确认后才停止**：声学VAD只触发确认模式，不直接停止TTS
- **区分语气助词和噪声**：避免"嗯"、"啊"、"咳嗽"等误触发打断
- **继续接收输入**：停止TTS后继续接收用户完整句子，再进入LLM

#### 流式输出优化（v3.2.3新增）

```
┌─────────────────────────────────────────────────────────────────┐
│                    v3.2.3 流式输出流程                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  LLM流式输出文本块                                              │
│       │                                                         │
│       ├──────→ 发送到前端实时显示                               │
│       │                                                         │
│       └──────→ 发送到 StreamingTTSProcessor                     │
│                     │                                           │
│                     ├── 检测句子结束符？──→ 立即TTS转换         │
│                     │                                           │
│                     └── 积累达到80字符？──→ TTS转换             │
│                              │                                  │
│                              ▼                                  │
│                     音频块发送到前端播放                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**关键改进**：
- **实时显示**：LLM输出的文本块立即发送到前端显示
- **按句子TTS**：StreamingTTSProcessor按句子分段转换语音
- **流畅播报**：每个句子完成后立即发送音频块播放

```python
async def _stop_tts_for_interrupt(self):
    """
    停止TTS播报（语义VAD确认有效人声后调用）

    v3.5: 统一处理 THINKING 和 SPEAKING 状态的打断
    - 只有语义VAD确认有效人声后才调用此方法
    - 停止TTS、LLM、清空音频队列
    """
    if self._tts_stopped_for_interrupt:
        return

    self._tts_stopped_for_interrupt = True
    self._should_stop_llm = True  # 标记停止LLM输出

    # 停止流式TTS处理器
    if self._streaming_tts:
        self._streaming_tts.stop()
        self._streaming_tts = None

    # 取消LLM流式输出任务
    if self._llm_task and not self._llm_task.done():
        self._llm_task.cancel()
        self._llm_task = None

    # 通知前端清空音频队列
    await self._notify_clear_audio()
```

### 6.2 流式TTS处理（v3.3/v3.4 优化）

```python
class StreamingTTSProcessor:
    """
    流式TTS处理器 v3.3/v3.4

    关键改进：
    - 使用 asyncio.Lock() 确保顺序处理
    - 先进先出队列，避免音频乱序
    - 句子编号追踪，便于调试
    - v3.4: 消费者任务独立处理
    """

    async def start(self):
        """启动消费者任务"""
        self._consumer_task = asyncio.create_task(self._consumer_loop())

    async def _consumer_loop(self):
        """消费者循环 - 独立处理文本队列"""
        while self._is_running:
            text = await self._text_queue.get()
            await self._process_text(text)

    async def add_text(self, text: str):
        async with self._lock:  # 获取锁，确保顺序
            # 累积文本，按句子分段转换
            await self._synthesize_buffer_locked()

    # 分段策略
    SENTENCE_ENDINGS = ['。', '！', '？', '；', '\n']
    MAX_CHUNK_SIZE = 80
    MIN_CHUNK_SIZE = 5
```

**处理流程：**
```
LLM输出文本块1 → add_text() → 获取锁 → 转换句子1 → 发送音频1 → 释放锁
LLM输出文本块2 → add_text() → 等待锁 → 获取锁 → 转换句子2 → 发送音频2
LLM输出文本块3 → add_text() → 等待锁 → 获取锁 → 转换句子3 → 发送音频3
```

## 十一、总结

### 11.1 v3.5 核心特性

1. **并发架构**：音频接收与处理解耦，非阻塞架构
2. **LLM后台任务**：LLM推理不阻塞音频接收
3. **THINKING状态打断**：允许在LLM推理期间打断
4. **语义驱动打断**：不等声学静音阈值，语义VAD判断有效人声后立即停止TTS
5. **区分语气助词和噪声**：避免"嗯"、"啊"、"咳嗽"等误触发打断
6. **继续接收输入**：停止TTS后继续接收用户完整句子
7. **流式处理**：ASR流式输出 + 语义VAD流式判断
8. **流式输出**：LLM流式输出 + TTS实时播报
9. **并行处理**：语义VAD和情绪识别并行执行
10. **上下文连贯**：多轮对话支持上下文
11. **时延监控**：实时追踪各模块处理时延

### 11.2 关键优化

| 优化点 | 效果 |
|--------|------|
| 并发架构 (v3.5) | 音频接收不再阻塞，实时性更好 |
| THINKING打断 (v3.5) | 允许在LLM推理期间打断，用户体验更好 |
| 语义驱动打断 | 打断响应时间从>400ms降到<200ms |
| 语气助词过滤 | 避免误触发打断 |
| 噪声过滤 | 咳嗽等噪声不触发打断 |
| 流式ASR | 更快开始识别 |
| 流式语义VAD | 边说边判断，减少等待 |
| 并行处理 | 延迟降低 30-50% |
| LLM流式输出 | 用户无需等待完整响应 |
| 流式TTS | 按句子播报，体验更流畅 |
| TTS顺序保证(v3.3) | 使用异步锁确保音频按顺序播放 |
